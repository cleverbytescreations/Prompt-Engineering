## Reference / Master Data Cache Pattern

Include this section only when the project serves quasi-static reference/master data
(lookup tables, category/config tables, sequence configs, enums-in-DB) via read-heavy
`GET` endpoints AND a cache (Redis/equivalent) is available. Omit it entirely if the project

All read endpoints for quasi-static reference/master data MUST use a read-through cache
helper (e.g. `app/core/cache.py`); never hit [DB] on every request for data that almost never
changes, and never import the cache client (`[redis]`) directly in service files. Go through
the helper (`cache_get` / `cache_set` / `cache_delete`).

### Freshness - two layers, BOTH required
1. Explicit invalidation on write - the service calls `cache_delete` for every affected key
   variant AFTER the DB commit, whenever an admin/screen mutates the data (create / update /
   deactivate). Primary mechanism: users see fresh data immediately after a screen edit.
2. TTL safety-net / eviction - every `cache_set` passes `ttl_sec=[settings.<ttl_setting>]`
   (default [N] min, env-overridable via `[TTL_ENV_VAR]`) so a missed write path (direct DB
   patch, bulk import) self-heals and unused entries are evicted after the window rather than
   living indefinitely.

Neither layer alone is sufficient: TTL alone leaves data stale until expiry after a screen
edit; invalidation alone leaves a missed write path stale forever.

### Rules
- One cache key per filter variant (e.g. `[prefix]:{type}:all`, `[prefix]:{type}:active`)
- Serialize via the Pydantic output schema using `model_dump(mode="json")` / `model_validate()`
  — plain `model_dump()` leaves `datetime`/`UUID` fields as Python objects, which `json.dumps`
  cannot serialize (`TypeError: Object of type datetime is not JSON serializable`); `mode="json"`
  converts them to JSON-compatible strings first
- `cache_delete` ALL key variants of the affected type AFTER `[db.commit()]`
- The in-process fallback must be TTL-aware (e.g. `time.monotonic()`); multi-worker prod
  MUST configure [Redis]
- Include a canonical read-through + invalidate-on-write service code block using the
  project's actual session style and helper names.

## Append-Only / Time-Series Table Patterns

[INCLUDE THIS SECTION when the project has any append-only table (audit log, notification log,
 event log, backup log, or similar) expected to grow without bound over time. OMIT entirely if
 no such table exists in the schema — do not emit a dead section.]

Apply these patterns to every append-only, time-ordered table in the project.

### Index choice: BRIN, not B-Tree, on `created_at`

Any append-only table with `server_default=now()` and no out-of-order inserts must use
a **BRIN** index on `created_at` instead of B-Tree:
```sql
CREATE INDEX idx_[table]_created_brin ON [table] USING BRIN (created_at);
```
BRIN stores only min/max per 128-page block range — ~8–12× less disk/write overhead
than B-Tree with equivalent range-scan coverage. Fall back to B-Tree only when
out-of-order inserts are possible.

### Retention configuration pattern

Expose the retention window as a settings/config env var; compute the cutoff at
call-time, never store it:
```python
[table]_retention_days: int = Field(default=[N])  # e.g. 2555 = 7 years; 0 = disabled
cutoff = datetime.now(tz=timezone.utc) - timedelta(days=settings.[table]_retention_days)
```

### Repository: `count_expired` / `delete_expired` (no commit in repo)

```python
def count_expired(db: Session, cutoff: datetime) -> int: ...   # dry-run safe
def delete_expired(db: Session, cutoff: datetime) -> None: ... # service commits
```

### Service: atomic delete + audit trace

Perform `count_expired` → `delete_expired` → `write_audit(action="PURGE_[TABLE]")` →
`db.commit()` in a **single transaction**. If the commit fails, neither the deletion nor
the evidence of it persists. Always support a `dry_run: bool = False` flag.

### API endpoint: admin-only with `?dry_run`

Gate behind an admin-only permission. Return a consistent envelope:
```json
{"purged_count": N, "cutoff_before": "ISO-datetime", "dry_run": false, "skipped": false}
```

### Cron script (`[backend_dir]/scripts/purge_[table].py`)

Standalone script reading the same env vars as the API. Must accept `--dry-run`,
exit `0` on success, `1` on error.
Suggested monthly schedule: `0 2 1-7 * 0  docker compose exec -T api python scripts/purge_[table].py`

Reference `architecture-deep-dive` skill — Append-Only Table Scale Strategy for
when to escalate to `pg_partman` monthly partitioning (trigger: 2 GB or 2–5M rows).

## File 5: `.agents/hooks/prefer_lsp.py`

Adapt the extension list to the project's primary typed languages.
Keep the hook lightweight because it runs frequently.

```python
#!/usr/bin/env python3
import json
import sys

data = json.load(sys.stdin)
tool_input = data.get("tool_input", {}) or {}

candidates = []
for key in ("path", "paths", "glob", "pattern", "query", "include"):
    value = tool_input.get(key)
    if isinstance(value, str):
        candidates.append(value)
    elif isinstance(value, list):
        candidates.extend(str(v) for v in value)

text = " ".join(candidates).lower()

lsp_exts = ("[.ext1]", "[.ext2]")  # e.g. (".ts", ".tsx", ".py")

if any(ext in text for ext in lsp_exts):
    print(json.dumps({
        "hookSpecificOutput": {
            "hookEventName": "PreToolUse",
            "permissionDecision": "deny",
            "permissionDecisionReason": (
                "For typed-code navigation, use the LSP tool instead of Grep. "
                "Prefer definitions, references, symbols, hover/type info, and diagnostics. "
                "Use Grep only as a fallback when LSP cannot answer."
            )
        }
    }))

sys.exit(0)
```

#!/bin/bash
# Runs tsc --noEmit after every Edit/Write on a .ts/.tsx file.
# Exit non-zero when errors are found so Codex surfaces them immediately.

INPUT=$(cat)

if ! command -v jq &>/dev/null; then
  exit 0
fi

FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty' 2>/dev/null)

if [[ ! "$FILE" =~ \.(ts|tsx)$ ]]; then
  exit 0
fi

FRONTEND_DIR="./[frontend_dir]"
if [ ! -d "$FRONTEND_DIR" ]; then
  exit 0
fi

DIAG=$(cd "$FRONTEND_DIR" && npx tsc --noEmit --pretty false 2>&1)
TSC_EXIT=$?

if [ $TSC_EXIT -eq 0 ]; then
  exit 0
fi

echo "TypeScript errors detected after editing: $FILE"
echo ""
echo "$DIAG" | head -50
exit 2
```

**For Python-only projects:** replace `tsc` with `mypy [module] --ignore-missing-imports`.
**For Go/Rust/Java/etc.:** replace with the language's type checker or build command.

Special rule for `run-tests.md`:
- Detect whether the project root has a Docker Compose file (`docker-compose.yml`, `docker-compose.yaml`, `compose.yml`, or `compose.yaml`).
- If Docker Compose exists and defines the app/test service, generate unit-test and suite commands that run inside that container, for example `docker compose exec [service] [test_runner] ...`.
- Include a container readiness check such as `docker compose ps [service]` and a project-appropriate `docker compose up -d [service]` instruction when needed.
- If no Docker Compose-backed test service exists, generate standalone host/in-project test commands instead.
- Do not include both Docker and standalone command variants unless the discovered project genuinely supports both.
- If Python/pytest with SQLite is detected, prefer in-memory SQLite test fixtures and include `python -m pytest ... -p no:cacheprovider`; do not generate file-backed SQLite migration probes.
- If Python coverage is requested or coverage tooling is detected, include a focused `--cov` command that writes coverage data inside the repo or a known writable temp directory.
- If Alembic/PostgreSQL-specific behavior is detected, instruct Codex to verify migrations with Docker/Postgres rather than SQLite.

# /fix-error

Efficiently diagnose and fix a specific error without reading unrelated code.

## Argument
Paste the error/traceback after the command: `/fix-error <traceback>`

## Instructions

### Step 1 - Parse the error
Extract the exact file and line number, error type and message, failing symbol, and any chained cause.

### Step 2 - Read only what is needed
- Read only a small window around the failing line.
- Use LSP hover/type info on the failing symbol when available.
- Use LSP definition/references only when the error depends on a declaration or shared utility.
- If the error is in a dependency, find the last stack frame that belongs to this project.

### Step 3 - Fix
Apply the minimal safe change. Do not refactor surrounding code unless the root cause requires it.

### Step 4 - Verify
Run the narrowest relevant test or checker:
```bash
[test_cmd] <specific_test_or_file>
```

### Step 5 - Report
State what was wrong, what changed, and the file:line reference.
```

### `migrate.md` blueprint
```markdown
# /migrate

Generate and apply a `[migration_tool]` migration.

## Argument
Pass migration name: `/migrate <name>`

## Instructions

1. Check current migration state:
   ```bash
   cd [backend_dir] && [migration_tool] current
   ```

2. Generate/create the migration:
   ```bash
   cd [backend_dir] && [migration_tool] [autogenerate_cmd] -m "$ARGUMENT"
   ```

3. Review the generated migration before applying:
   - Confirm upgrade/apply creates only expected objects.
   - Confirm downgrade/rollback reverses the change safely when the tool supports it.
   - Confirm tenant/platform schema rules if this project has schema separation.
   - Confirm migrations connect through the direct PostgreSQL path rather than PgBouncer/poolers when the discovered architecture requires that.
   - Confirm no destructive data loss unless explicitly requested.

4. Apply:
   ```bash
   cd [backend_dir] && [migration_tool] upgrade head
   ```

5. Verify:
   ```bash
   cd [backend_dir] && [migration_tool] current
   ```

6. If migration fails, do not edit the database manually. Fix the model/migration and rerun the migration workflow.
```
