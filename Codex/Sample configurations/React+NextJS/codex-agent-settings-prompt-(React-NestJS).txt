# Prompt: Generate Codex Agent Configuration for a React + NestJS Fastify + PostgreSQL Project

You are setting up Codex configuration for an existing software project.

The target stack family is:
- UI: React
- API: Node.js with NestJS configured to use the Fastify adapter
- Database: PostgreSQL

Do not assume the exact project shape. The project may be a monorepo, a pair of sibling apps, or a single package with frontend and backend folders. Every generated file must be based on what you discover in the repository.

Never generate `.claude/` files. Never use `CLAUDE.md` as the primary routing document. Generate only Codex-supported repository resources such as `AGENTS.md`, `.agents/skills`, `.agents/hooks`, `.agents/commands`, and `.agents/settings.json` when supported by the current Codex setup.

---

## Discovery Requirements

Before generating anything, inspect the project and derive every value from the repository:

1. Read `AGENTS.md` if it exists; otherwise read `README.md` and root-level docs.
2. List the root directory to identify top-level apps, packages, docs, and tooling.
3. Read root package manifests and workspace files:
   - `package.json`
   - `pnpm-workspace.yaml`
   - `turbo.json`
   - `nx.json`
   - `lerna.json`
   - lockfiles such as `pnpm-lock.yaml`, `package-lock.json`, `yarn.lock`, or `bun.lockb`
4. Identify the frontend package or folder by inspecting package dependencies and source entry points.
5. Identify the backend package or folder by inspecting package dependencies and NestJS entry points.
6. Confirm whether NestJS uses the Fastify adapter by checking for packages and code such as:
   - `@nestjs/platform-fastify`
   - `NestFactory.create<NestFastifyApplication>()`
   - `new FastifyAdapter(...)`
7. Identify exact frontend libraries:
   - React version
   - build framework or tool such as Vite, Next.js, Remix, or custom tooling
   - router
   - data-fetching library
   - state library
   - form library
   - validation library
   - UI library
   - styling system
   - mock/demo tooling if any
   - i18n library if any
8. Identify exact backend libraries:
   - NestJS version
   - Fastify adapter version
   - ORM, query builder, or PostgreSQL client
   - migration tool
   - request validation approach
   - auth strategy
   - queue/background worker system
   - cache client
   - logging and observability tools
   - test runner
   - linter and formatter
9. Identify data and infrastructure:
   - PostgreSQL version if declared
   - Redis or other cache if present
   - queue broker if present
   - object storage if present
   - search engine if present
   - Docker Compose, Dockerfiles, CI, deployment, and observability config
10. Check whether `Docs/`, `docs/`, or equivalent planning folders exist and list their contents.
11. Find the task checklist file if one exists, commonly `Docs/tasks.md`, `Docs/*Checklist*.md`, `docs/tasks.md`, or similar.
12. Inspect whether `.agents/settings.json` already exists and preserve the compatible shape if it does.
13. Inspect whether `.agents/skills/`, `.agents/hooks/`, and `.agents/commands/` already exist.
14. Confirm the absolute path to the backend directory and the relative path to the frontend directory for hook generation.
15. Confirm the package manager and workspace command style before writing command prompts.

Every detail must come from what you found. Never use placeholder project names, never copy content from another repository, and never leave unresolved placeholders in generated files.

## Backend Architecture Rule

Follow the discovered NestJS layering. If the repo already has a clear pattern, preserve it.

Default layering when compatible with the repo:
1. Module layer - `[modules_path]` - groups domain providers, controllers, imports, and exports
2. Controller layer - `[controllers_path]` - HTTP concerns only: route decorators, guards, pipes, request DTOs, response mapping
3. Service layer - `[services_path]` - business logic, orchestration, transactions, state changes, side effects
4. Repository/data-access layer - `[repositories_path]` - database query construction and persistence
5. Model/entity/schema layer - `[models_path]` - ORM models, entities, Prisma schema, Drizzle schema, or table definitions

Controller -> Service -> Repository/Data Access -> Database.
No layer imports from a layer above it.

## NestJS Module Rules

- Keep each domain inside the discovered module structure.
- Export only providers that other modules actually need.
- Prefer dependency injection over manually instantiating services, repositories, clients, or queues.
- Keep module registration explicit and aligned with existing `imports`, `providers`, `controllers`, and `exports`.
- Use dynamic modules only when configuration or external integration setup requires them.

## Controller Rules

- Controllers own HTTP decorators, route paths, guards, pipes, interceptors, status codes, and request/response mapping.
- Controllers call services and avoid business logic.
- Controllers do not perform direct database calls.
- Controllers do not enqueue jobs directly unless the repo already uses controllers as orchestration boundaries; otherwise queue handoff belongs in services or outbox workers.
- Use the project's discovered auth guard and role/permission pattern.
- Use explicit response status decorators or Fastify reply behavior only when consistent with existing code.
- Avoid leaking ORM entities directly if the project uses response DTOs or serializers.

## Fastify Adapter Rules

- Bootstrap must use the discovered Fastify adapter pattern, such as `NestFactory.create<NestFastifyApplication>(AppModule, new FastifyAdapter(...))`.
- Use Fastify-specific request/reply types only where the existing project needs platform-specific behavior.
- Do not introduce Express middleware, Express request types, or Express-only libraries unless the project already bridges them intentionally.
- Prefer NestJS guards, pipes, filters, and interceptors before raw Fastify hooks unless the repo already centralizes behavior in Fastify plugins.
- When adding file upload, cookies, compression, CORS, helmet, or multipart behavior, use packages compatible with NestJS on Fastify and verify registration order.

## DTO / Validation Rules

- Use the validation style already installed and used by the repo.
- If using `class-validator`, keep DTOs explicit and ensure `ValidationPipe` settings are respected.
- If using Zod/TypeBox/Joi, keep request and response schemas close to the existing contract pattern.
- Do not claim runtime validation exists unless configured.
- Keep internal persistence fields out of public response DTOs.

## Service Layer Rules

- Services own business rules, orchestration, workflow transitions, transaction boundaries, and integration handoffs.
- Services receive dependencies through NestJS dependency injection.
- Services delegate database query construction to repositories/data-access providers when that pattern exists or is introduced for new domains.
- Services may coordinate multiple repositories inside one transaction using the detected transaction API.
- Services should raise domain/application errors that are mapped by filters or controller-level handling according to the repo pattern.
- Services should not contain HTTP decorator logic, raw request parsing, or response-shaping code beyond domain return values.

## Repository / Data-Access Rules

- Use the detected data-access style:
  - Prisma: repositories receive `PrismaClient` or transaction client through an injected database service
  - TypeORM: repositories use injected `Repository<T>`, `DataSource`, or transaction manager
  - Drizzle: repositories receive the Drizzle database or transaction object
  - Knex/node-postgres: repositories receive the Knex instance, transaction, pool, or client
- Repositories own query construction, filters, joins, pagination, aggregates, persistence helpers, and result mapping.
- Repositories contain no business workflow decisions, authorization decisions, queue dispatch, or HTTP concerns.
- Repositories return typed entities, DTO-ready objects, scalars, `null`, or empty arrays according to the repo's established style.
- Keep transactions opened by services or unit-of-work helpers, not hidden inside low-level repository calls unless the repo's pattern does that deliberately.

Canonical signatures should be adapted to the detected database tool:

```typescript

## Security Rules

- Use the discovered authentication guard, authorization decorator, and role/permission pattern.

- Never log secrets, tokens, passwords, or sensitive PII.

### Hook 1: `.agents/hooks/prefer_lsp.py`

Use this PreToolUse hook to block broad grep/search tool use when the query targets typed source files. Codex should use LSP first for symbol navigation, then use targeted search only for config strings, comments, TODOs, and non-symbol literals.

```python
#!/usr/bin/env python3
import json
import sys

data = json.load(sys.stdin)
tool_input = data.get("tool_input", {}) or {}

candidates = []
for key in ("path", "paths", "glob", "pattern", "query", "include"):
    value = tool_input.get(key)
    if isinstance(value, str):
        candidates.append(value)
    elif isinstance(value, list):
        candidates.extend(str(v) for v in value)

text = " ".join(candidates).lower()

lsp_exts = (".ts", ".tsx", ".js", ".jsx")

if any(ext in text for ext in lsp_exts):
    print(json.dumps({
        "hookSpecificOutput": {
            "hookEventName": "PreToolUse",
            "permissionDecision": "deny",
            "permissionDecisionReason": (
                "For typed JavaScript/TypeScript navigation, use LSP before Grep. "
                "Prefer definitions, references, symbols, hover/type info, and diagnostics. "
                "Use Grep for config strings, comments, TODOs, and non-symbol literals."
            )
        }
    }))

sys.exit(0)
```

Include `.py` only if Python is actually present as application code.

### Hook 2: `.agents/hooks/ts-diagnostics.py`

Prefer a Python hook for cross-platform Windows compatibility. The hook should:
- read Codex tool JSON from stdin
- inspect edited file path
- run the detected TypeScript diagnostics command for the relevant package
- cap output to a small number of lines
- exit non-zero only when the project wants hard diagnostic failures

Template:

```python
#!/usr/bin/env python3
import json
import os
import subprocess
import sys
from pathlib import Path

ROOT = Path(__file__).resolve().parents[2]
FRONTEND_DIR = ROOT / "[frontend_dir]"
BACKEND_DIR = ROOT / "[backend_dir]"

FRONTEND_TYPECHECK = [[frontend_typecheck_argv]]
BACKEND_TYPECHECK = [[backend_typecheck_argv]]

def main() -> int:
    try:
        data = json.load(sys.stdin)
    except Exception:
        return 0

    file_path = (
        data.get("tool_input", {}).get("file_path")
        or data.get("tool_input", {}).get("path")
        or ""
    )
    if not file_path.endswith((".ts", ".tsx")):
        return 0

    path = Path(file_path)
    abs_path = path if path.is_absolute() else ROOT / path

    if FRONTEND_DIR in abs_path.parents or abs_path == FRONTEND_DIR:
        cwd = FRONTEND_DIR
        cmd = FRONTEND_TYPECHECK
    elif BACKEND_DIR in abs_path.parents or abs_path == BACKEND_DIR:
        cwd = BACKEND_DIR
        cmd = BACKEND_TYPECHECK
    else:
        return 0

    if not cwd.exists():
        return 0

    result = subprocess.run(cmd, cwd=str(cwd), text=True, capture_output=True)
    if result.returncode == 0:
        return 0

    output = (result.stdout + "\n" + result.stderr).strip().splitlines()
    print(f"TypeScript diagnostics failed after editing: {file_path}")
    print("\n".join(output[-80:]))
    return 2

if __name__ == "__main__":
    raise SystemExit(main())
```

Replace `[frontend_typecheck_argv]` and `[backend_typecheck_argv]` with Python argv list contents, not shell strings. For example: `"pnpm", "--filter", "web", "run", "typecheck"`.

If the project has a root workspace typecheck command that is fast enough, use that instead of package-specific commands.
