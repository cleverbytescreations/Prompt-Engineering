You are bootstrapping the Claude config for a MOBILE app project (native
Android or native iOS). The set of skills, commands, and hooks you create
MUST be derived from this repo's ACTUAL stack and ACTUAL docs — do not assume
any specific platform, libraries, features, or domain until you have detected
them in Step 0–1.

# Prime directive
Every artifact you produce is judged on one metric: tokens-per-successful-task.
A smaller, sharper config beats a larger, thorough one. When in doubt, omit.

The three jobs this config must make cheap and reliable:
  J1. DEFECT WORK  — locate the file/line behind a bug or crash fast, with
      minimal search, then propose a minimal patch.
  J2. FEATURE WORK — add a new screen/feature matching existing conventions
      using the fewest reads and the least guessing.
  J3. PORTABILITY  — when re-run in a fresh mobile repo that ships an
      architecture and/or use-case doc, bootstrap a correct config by
      reading those docs FIRST (they are a pre-built capability map) instead
      of blind code scanning.

Five failure modes to avoid above all else:
  1. Ignoring existing docs. If the repo has an architecture / use-case /
     design doc, reading code to re-derive what the doc already states is
     pure token waste. Docs are the FIRST discovery source (Step 0).
  2. Blanket file-level denies (e.g. `Read(<file>)`) that block every
     legitimate read of a file because ONE field in it is sensitive.
     Always protect at the FIELD level, never the FILE level.
  3. Recursive greps that wander into generated/derived directories
     (Android: build/, .gradle/, .idea/, out/; iOS: DerivedData/, Pods/,
     .build/, *.xcarchive, .swiftpm/) and scan thousands of files per call.
     These burn 20–90s and return only compiled noise.
  4. Pattern-only secret denies (e.g. `Bash(grep*storePassword*)`) that match
     the grep PATTERN string and block legitimate security-audit commands like
     `grep -r "storePassword" app/src/`. Always scope secret denies to the
     TARGET FILE, not just the field name:
       WRONG: "Bash(grep*storePassword*)"
       RIGHT: "Bash(grep*storePassword*build.gradle.kts*)"
     Cover unscoped searches in the guard-build-dir-search.sh hook instead.
  5. Assuming the wrong platform. Android-only assumptions (gradlew, build/)
     break an iOS repo; iOS-only assumptions (xcodebuild, DerivedData) break
     an Android repo. Detect the platform in Step 1 and branch every
     platform-specific artifact accordingly.

# Step 0 — Ingest existing docs FIRST (highest leverage, do before code)

Before reading any source, locate and read the project's own documentation —
this is the cheapest, densest source of the Capability Map.

Discover docs (read-only, parallelized; summarize internally, do NOT echo raw):
- `git ls-files | grep -iE '\.(md|markdown|txt|adoc|rst)$'`  (cap 200 lines)
- Prioritize, in order:
    1. Architecture / design docs: paths or names matching
       *architectur*, *design*, *system*, *overview*, ADRs under docs/adr/.
    2. Use-case / requirements / spec docs: *use-?case*, *requirement*,
       *spec*, *feature*, *PRD*, *user-?stor*.
    3. Contributor / onboarding docs: README*, CONTRIBUTING*, docs/, Docs/.
    4. Any existing CLAUDE.md (do not overwrite without asking — Step 6).
- Read the top 3–6 most relevant docs in full (they are usually small).

From the docs, pre-fill as much of the Capability Map (Step 1) as you can,
and record:
  - the declared module / package layout and the "source of truth" components
  - which components are explicitly called LEGACY / dead / deprecated
  - explicit invariants, risks, coupling notes, and "do not do X" statements
  - any architecture-index doc path (Step 3 will cite it in CLAUDE.md so
    future sessions read it before exploring)

Treat the docs as authoritative for INTENT, but verify load-bearing claims
against code in Step 1 (docs drift). When a doc and the code disagree, trust
the code and note the drift as a pain_point.

If NO architecture/use-case doc exists, say so and fall back to code-only
discovery — but still scan README/CONTRIBUTING before source.

# Step 1 — Detect platform, then discover code (read-only, parallelized)

## 1a. Platform detection (decides every branch below)
Run in parallel and classify:
- Android signals: `gradlew`, `*.gradle(.kts)`, `settings.gradle(.kts)`,
  `gradle/libs.versions.toml`, `AndroidManifest.xml`, `*.kt` / `*.java`.
- iOS signals: `*.xcodeproj`, `*.xcworkspace`, `Package.swift`, `Podfile`,
  `project.pbxproj`, `*.xcconfig`, `Info.plist`, `*.swift`.
- Cross-platform signals (note but keep native focus): `pubspec.yaml`
  (Flutter), `package.json` + `android/` + `ios/` (React Native),
  Kotlin Multiplatform (`kotlin("multiplatform")`).

Set `platform` = android | ios | cross-platform(<framework>). If BOTH native
toolchains are present (e.g. an RN repo with android/ and ios/), treat the
JS/Dart layer as primary and generate native sub-skills only if those folders
contain hand-written native code.

## 1b. Code discovery (summarize internally, do NOT echo raw output)
Common to all platforms:
- `git ls-files`  (cap 500 lines; if larger, sample by directory)
- `git log --oneline -10 --name-only`  (which files actively change; used in
  Step 5 to resolve ties between coexisting patterns)
- Read any CLAUDE.md found in Step 0
- Sample 5–10 representative source files from the largest packages/targets

Android-specific:

- Read: root build.gradle(.kts), settings.gradle(.kts),

iOS-specific:

# Step 2 — Decide what to generate (the dynamic part)

Each generated artifact must justify its tokens.

Skill rules:
- Generate a skill ONLY for a slot with a concrete value AND non-trivial
  conventions in this repo. UNKNOWN slots get NO skill.
- One skill per **concern**, not per library. Merge thin skills.
- Hard cap: 5 skills total. If you'd exceed it, merge or drop the weakest.
- Every "special" capability (camera, payments, ML, BLE, etc.) detected in
  code gets its own skill — highest leverage.
- Each skill ≤ 150 lines, with concrete file:line citations from THIS repo.
- ALWAYS generate one platform skill covering build/dependency/signing ops
  (Android: gradle + version catalog; iOS: xcodebuild/SPM/CocoaPods +
  signing). It anchors J1/J2 build steps.

Command rules — generate for **observed workflows**, not generic ones:
- Always include:
    `find-feature` — entry → screen → state holder → data source chain
                     (the J1/J2 starting point; outputs a path chain only).
    `fix-crash`    — stack trace / crash log → ranked hypotheses with
                     file:line → minimal patch.
- Conditionally include based on Capability Map:
    Compose            → `add-screen` (Compose scaffold the repo uses)
    SwiftUI            → `add-screen` (SwiftUI View + state scaffold)
    UIKit              → `add-screen` (ViewController/coordinator scaffold)
    Views(XML)         → `add-fragment`
    multi-module / multi-target → `module-deps`
    multi-flavor / multi-scheme → `flavor-switch` / `scheme-switch`
    Room / CoreData / SwiftData → `add-entity` (entity + DAO/model + migration)
    Retrofit/Ktor/URLSession/Alamofire → `add-endpoint`
    CI present         → `ci-repro` (run the same checks locally)
- Drop any command whose workflow isn't actually used here.

Hook rules — only propose hooks with a clear token/time payoff:
- PreToolUse Bash: block expensive whole-project builds; steer to the
  narrowest task observed in CI.
    Android → block bare `./gradlew build`; suggest `:app:assembleDebug`.
    iOS     → block bare `xcodebuild` clean-build-all; suggest
              `xcodebuild -scheme <S> -configuration Debug -sdk
              iphonesimulator build` (or `swift build` for SPM-only).
- PreToolUse Bash: block grep/find targeting generated/derived dirs — re-route
  to `git grep`. ALWAYS GENERATE (every mobile repo has such dirs).
- PreToolUse Read: if `secrets` is non-empty, emit a field-level reminder when
  the secret-bearing file is read. Advisory (exit 0).
- PostToolUse Edit on source files: run formatter/linter ONLY if one is
  configured (Android: ktlint/detekt/spotless; iOS: SwiftLint/SwiftFormat) —
  scoped to the changed file. Otherwise defer to `_proposed_hooks`.
- UserPromptSubmit: if the prompt names a screen/feature word found in the file
  tree, suggest `/find-feature <name>` before any exploration.
- Do NOT propose hooks for tools not present in this repo.

Rules for the `permissions` block:

PRECEDENCE: In Claude Code, when a command matches BOTH allow and deny,
DENY WINS. The whole design relies on this:
  allow "Bash(grep*build.gradle.kts*)"                  ← safe greps pass
  deny  "Bash(grep*storePassword*build.gradle.kts*)"    ← signing greps blocked
Both match `grep storePassword app/build.gradle.kts`; deny wins, so extraction
is blocked while ordinary config greps still pass. Generators that flatten this
into a single rule will break the design.

BASENAME DERIVATION: For each `secrets` entry, compute
  <secret-file-basename> = basename of the path, e.g.
    "app/build.gradle.kts"        → "build.gradle.kts"
    "local.properties"            → "local.properties"
    "MyApp.xcodeproj/project.pbxproj" → "project.pbxproj"
    "GoogleService-Info.plist"    → "GoogleService-Info.plist"
    "Config.xcconfig"             → "Config.xcconfig"
Use the BASENAME (not the full path) in deny/allow globs — permission rules are
glob-matched against the literal command string, and basenames give the widest
correct match (covers ./relative and absolute paths Claude may use).

PROTECT SECRETS AT THE FIELD LEVEL, NOT THE FILE LEVEL. A blanket
`Read(<file>)` deny blocks every legitimate read (buildConfig, SDK levels,
deps, schemes, capabilities) and forces every metadata task to fail or prompt.
Do not do this.

For every `secrets` entry:
- Add the path to `allow` in BOTH forms: "Read(<path>)" and "Read(./<path>)".
- Add `Bash(grep*<basename>*)` allow so safe grep on the file works.
- For each sensitive field, add a deny SCOPED TO THE SECRET FILE:
    "Bash(grep*<field-name>*<secret-file-basename>*)"
  NOT "Bash(grep*<field-name>*)" ← blocks source-tree audits too.
  Glob order matters: field-name BEFORE file-name so it matches
  `grep storePassword build.gradle.kts` but NOT
  `grep -r storePassword app/src/...` (a valid audit that must pass).
- Add a narrow `cat` deny: "Bash(cat*<secret-file-basename>*)".
- Enable `guard-signing-read.sh` for the Read-tool path.
- Add a one-line Key Rule in CLAUDE.md citing the file by name.
- `guard-build-dir-search.sh` covers the gap: unscoped `grep -r <field> .`
  with no src/-scoped path is blocked there.

Allowlist the narrow build/test/run/git commands actually used in CI or docs,
plus `Bash(git grep*)` and (Android) `Bash(grep*libs.versions.toml*)` —
prompting for these on every search is pure token waste.

Conventional hook scripts to generate (use these exact filenames):

  guard-build.sh               ← PreToolUse on Bash. ALWAYS GENERATE.
                                 Android: blocks bare `./gradlew build`,
                                 steers to the narrowest CI task
                                 (assembleDebug). iOS: blocks
                                 whole-project `xcodebuild` clean builds,
                                 steers to a `-scheme … -configuration Debug`
                                 simulator build (or `swift build`). Exit 2
                                 on block, exit 0 otherwise. Branch on the
                                 build tool detected in the command string
                                 (`gradlew` vs `xcodebuild`/`swift`).

  guard-build-dir-search.sh    ← PreToolUse on Bash. ALWAYS GENERATE.
                                 Responsibilities:
                                 (a) hard-block grep/find targeting
                                     generated/derived dirs — Android:
                                     build/, .gradle/, .idea/, out/; iOS:
                                     DerivedData/, Pods/, .build/,
                                     *.xcarchive, .swiftpm/ — emit a
                                     `git grep` replacement in stderr;
                                 (b) re-route grep for generated build
                                     constants (Android BuildConfig; iOS
                                     R.swift/SwiftGen/generated xcconfig) to
                                     the real source of truth;
                                 (c) hard-block grep for secret field names
                                     (populate from the `secrets` slot) when
                                     the command has NO src/-scoped path;
                                     allow when path contains src/, main/java,
                                     main/kotlin, Sources/, or the app target
                                     dir (valid security audit);
                                 (d) warn-only on unscoped `grep -r` with no
                                     source path.
                                 Typical savings: prevents 20–90s scans of
                                 thousands of generated files per misfired
                                 search, plus covers the file-scoped deny gap.

                                 The deny rules enforce the hard stops; this

  suggest-find-feature.sh      ← UserPromptSubmit. ALWAYS GENERATE if a

                                 Scans the prompt for screen/component/feature
                                 names found in the file tree; prints a
                                 one-line hint. Always exit 0.

  lint-changed-<lang>.sh       ← PostToolUse on Edit. GENERATE ONLY if a

                                 ktlint/detekt/spotless; iOS: SwiftLint/
                                 SwiftFormat). <lang> = kotlin | java | swift.
                                 If absent → goes to `_proposed_hooks`, script

Each generated script MUST start with:
    #!/usr/bin/env bash
    set -euo pipefail
…and end with explicit `exit 0` on the success path. Parse stdin JSON (rules
below) and match the exit-code contract. Name extra repo-specific hooks with
the `<verb>-<noun>.sh` shape so they sort and grep predictably.

Rules for the `hooks` block (active hooks):
- Reference scripts via $CLAUDE_PROJECT_DIR/.claude/hooks/<name>.sh — never
  relative paths.
- Write the script bodies this step, in .claude/hooks/, and `chmod +x` them.
  A hook entry without an executable script is a bug.
- Parse tool input from stdin JSON:
    payload="$(cat || true)"
    cmd="$(printf '%s' "$payload" | jq -r '.tool_input.command // empty')"
  Provide a sed fallback (jq is not guaranteed on every dev machine).
- For Read-tool hooks, parse `.tool_input.file_path` instead of `.command`.
- Exit codes: 0 = allow/no-op; 2 = block (stderr shown to Claude as the tool
  error). Never use exit 1 — reserved for script-internal errors.
- Non-blocking hooks (UserPromptSubmit hints, Read reminders, advisory checks)
  MUST always `exit 0`.
- Build-guard regex: match the build tool token anywhere in the command string
  (`gradlew`, `xcodebuild`, or `swift build`) — never a bare word-boundary
  anchor (the command arrives as `./gradlew …` / `xcodebuild …`).
- Build-dir-guard regex: match `(grep|find)` … followed by a generated-dir
  token for the detected platform.

Rules for the `_proposed_hooks` block (deferred):
- Any hook depending on a tool not present (ktlint/detekt/spotless absent;
  SwiftLint/SwiftFormat absent) goes here, NOT in active `hooks`.
- Each deferred entry MUST include a `_blocker` field naming the missing
  prerequisite, e.g. "_blocker": "SwiftLint not in Podfile/SPM; add it first."
- `_proposed_hooks` is informational only — Claude Code does not execute it.

Skeletons (adapt to platform + repo; do NOT ship verbatim):

  # guard-build.sh  (handles Android gradlew AND iOS xcodebuild/swift)
  #!/usr/bin/env bash
  set -euo pipefail
  payload="$(cat || true)"
  if command -v jq >/dev/null 2>&1; then
    cmd="$(printf '%s' "$payload" | jq -r '.tool_input.command // .command // empty')"
  else
    cmd="$(printf '%s' "$payload" | sed -n 's/.*"command"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/p' | head -n1)"
  fi
  # Android: block bare `./gradlew build`
  case "$cmd" in
    *"gradlew"*)
      case "$cmd" in
        *"assemble"*|*"install"*|*"test"*|*"lint"*|*"dependencies"*|*"--stop"*|*"clean"*) exit 0 ;;
      esac
      if printf '%s' "$cmd" | grep -Eq 'gradlew([[:space:]]+-[^[:space:]]+)*[[:space:]]+(:[a-zA-Z0-9_-]+:)?build([[:space:]]|$)'; then
        { echo "Blocked: \`./gradlew build\` triggers a full build.";
          echo "Prefer: ./gradlew :app:assembleDebug --offline"; } 1>&2
        exit 2
      fi
      exit 0 ;;
  esac
  # iOS: block whole-project xcodebuild without a scheme/target (clean build all)
  case "$cmd" in
    *"xcodebuild"*)
      if printf '%s' "$cmd" | grep -Eq '\-scheme[[:space:]]|\-target[[:space:]]|\-list|test'; then exit 0; fi
      { echo "Blocked: bare \`xcodebuild\` builds every target.";
        echo "Prefer: xcodebuild -scheme <Scheme> -configuration Debug -sdk iphonesimulator build"; } 1>&2
      exit 2 ;;
  esac
  exit 0


  # guard-build-dir-search.sh  (generated-dir token differs per platform)
  #!/usr/bin/env bash
  set -euo pipefail
  payload="$(cat || true)"
  if command -v jq >/dev/null 2>&1; then
    cmd="$(printf '%s' "$payload" | jq -r '.tool_input.command // .command // empty')"
  else
    cmd="$(printf '%s' "$payload" | sed -n 's/.*"command"[[:space:]]*:[[:space:]]*"\([^"]*\)".*/\1/p' | head -n1)"
  fi
  # 1. Hard-block grep/find into generated/derived dirs.
  #    Android token: (\.|/)?build(/|$)
  #    iOS token:     DerivedData|/Pods(/|$)|\.build(/|$)|\.xcarchive|\.swiftpm
  #    Populate gen_dirs at generation time for the detected platform.
  gen_dirs='(\.|/)?build(/|$)|DerivedData|/Pods(/|$)|\.build(/|$)|\.xcarchive|\.swiftpm|\.gradle|\.idea|/out(/|$)'
  if printf '%s' "$cmd" | grep -Eq "(grep|find)[[:space:]].*(${gen_dirs})"; then
    { echo "Blocked: searching generated/derived dirs is slow and returns only build artifacts.";
      echo "Use:  git grep '<pattern>' -- '<source-dir>'"; } 1>&2
    exit 2
  fi
  # 2. Re-route grep for generated build constants.
  #    Android → BuildConfig; iOS → R.swift / SwiftGen / generated xcconfig.
  if printf '%s' "$cmd" | grep -Eq "grep.*(BuildConfig|R\.generated|SwiftGen)"; then
    if printf '%s' "$cmd" | grep -Eq "src/|Sources/"; then exit 0; fi
    { echo "Blocked: grepping generated build constants searches generated code.";
      echo "For VALUES: read the build config / xcconfig / Info.plist directly.";
      echo "For USAGES: git grep '<symbol>' -- '<source-dir>'"; } 1>&2
    exit 2
  fi
  # 3. Hard-block grep for secret field names without a src/-scoped path.
  #    Populate from the `secrets` slot (Android + iOS field names).
  secret_fields="storePassword|keyPassword|storeFile|keyAlias|signingConfig|DEVELOPMENT_TEAM|PROVISIONING_PROFILE|CODE_SIGN_IDENTITY"
  if printf '%s' "$cmd" | grep -Eq "grep.*(${secret_fields})"; then
    if printf '%s' "$cmd" | grep -Eq "src/|app/src|main/java|main/kotlin|Sources/"; then exit 0; fi
    { echo "Blocked: unscoped grep for signing fields could read the secret config file.";
      echo "To audit source for leakage, scope to the source tree:";
      echo "  git grep 'storePassword\\|DEVELOPMENT_TEAM' -- '<source-dir>'"; } 1>&2
    exit 2
  fi
  # 4. Warn-only on unscoped recursive grep.
  if printf '%s' "$cmd" | grep -Eq "^grep -r|^grep --recursive"; then
    if ! printf '%s' "$cmd" | grep -Eq "(src/|Sources/|git grep)"; then
      echo "Warning: unscoped recursive grep scans generated dirs. Prefer: git grep '<pattern>' -- '<source-dir>'" 1>&2
    fi
  fi
  exit 0


  # guard-signing-read.sh  (path glob + field list from the `secrets` slot)
  #!/usr/bin/env bash
  set -euo pipefail
  payload="$(cat || true)"
  if command -v jq >/dev/null 2>&1; then

  fi
  # Match each secret-bearing basename. Android example shown; add iOS cases
  # (*project.pbxproj, *GoogleService-Info.plist, *.xcconfig) when present.
  case "$file_path" in
    *build.gradle.kts) fields="storeFile, storePassword, keyAlias, keyPassword" ;;
    *project.pbxproj)  fields="DEVELOPMENT_TEAM, PROVISIONING_PROFILE, CODE_SIGN_IDENTITY" ;;
    *GoogleService-Info.plist) fields="API_KEY, GCM_SENDER_ID, CLIENT_ID" ;;
    *) exit 0 ;;
  esac

    echo "  $fields";

  exit 0

      *"$kw"*) echo "[hint] Prompt mentions \"$kw\" — consider \`/find-feature ${kw}\` first."; exit 0 ;;
    esac
  done
  exit 0
