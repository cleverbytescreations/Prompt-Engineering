# Prompt: Generate Claude Code Configuration for a React + Node.js/Fastify + PostgreSQL Project

You are setting up Claude Code configuration for a software project from scratch.
Before generating anything, **read the project** to discover its actual structure:

1. Read `README.md` (or root-level docs) — domain, purpose, users
2. List the root directory to find top-level folders
3. Read `package.json` at the project root — identify workspace setup (monorepo or not)
4. Read `package.json` inside the backend/server/api directory — identify exact packages:
   Fastify version, ORM (Prisma vs Drizzle vs TypeORM), validation library (Zod vs TypeBox),
   queue system (BullMQ vs Bull vs pg-boss), test runner (Vitest vs Jest), linter (ESLint vs Biome)
   **ESLint version check (critical for hook generation):** inspect the `eslint` version field.
   - ESLint v8.x → uses `.eslintrc.*` config; hook command: `npx eslint <file> --ext .ts -q`
   - ESLint v9.x → uses `eslint.config.js` flat config; hook command: `npx eslint <file> -q`
     (the `--ext` flag is removed in v9 — including it causes a fatal error)
5. Read `package.json` in the frontend directory — confirm exact React, TanStack Query,
   Zustand, Tailwind, and Router versions
6. Identify: frontend dir name, backend dir name, PostgreSQL version, Redis version,
   object storage (MinIO / S3 / Azure Blob), i18n library if any
7. Check if `prisma/schema.prisma` exists, or identify the ORM schema file location
8. Check if a `Docs/` (or `docs/`) folder exists and list its contents
9. Find the task checklist file (commonly `Docs/tasks.md` or similar)
10. Confirm the **absolute path** to the backend `src/` directory (needed for ESLint hook)
11. Confirm the **relative path** to the frontend directory (needed for ts-diagnostics hook)

Then generate every file listed below. Every detail must come from what you found —
never use placeholder names, never copy content from another project.

2. **Service layer** — `[backend_dir]/src/services/<domain>.service.ts`

3. **Repository layer** — `[backend_dir]/src/repositories/<domain>.repository.ts`

## Transactional Outbox Pattern

[THIS IS THE CANONICAL LOCATION FOR THE FULL OUTBOX SPEC]
All domain events ([list: e.g. UserCreated, AssignmentCompleted, NotificationRequested]) must:

1. Write an `[outbox_table]` row within the **same** `db.$transaction()` block as the
   state change — never outside it
2. Never enqueue a BullMQ job directly from a route handler or service without the outbox

Outbox row shape:
```typescript
{ id, event_type, payload: JSON, status: 'PENDING', created_at, published_at? }
```

Outbox polling worker (`[backend_dir]/src/workers/outbox.worker.ts`, interval: [N]s):
- Reads rows WHERE status = 'PENDING' ORDER BY created_at LIMIT [N]
- Enqueues the corresponding BullMQ job
- Sets status → 'PUBLISHED' on success
- Sets status → 'FAILED' on error; retries up to [N] times
- Sets status → 'DEAD_LETTER' after max retries

This guarantees no lost jobs on API crash or Redis restart.

- Database URL: stored in `[.env variable name]`; never hardcoded

## Prisma Schema Rules

- All models use UUID primary keys: `id  String  @id  @default(uuid())`
- Every model has `createdAt DateTime @default(now())` and `updatedAt DateTime @updatedAt`
- Soft deletes: `deletedAt DateTime?` — never hard-delete user-facing records

- Never edit a `.sql` migration file after it has been applied to any environment

  `RENAME COLUMN` SQL instead of accepting the destructive default
- Test `prisma migrate reset` locally after every major schema change
- Production deployments use `prisma migrate deploy` — never `prisma migrate dev` in CI/prod
- Rollback: Prisma has no built-in `down` migration; document the manual reversal SQL
  in a comment at the top of every migration file

## Transaction Patterns

Use `db.$transaction(async (tx) => { ... })` when multiple writes must be atomic:

```typescript
// Canonical: state change + outbox row in one transaction
await db.$transaction(async (tx) => {
  const record = await domainRepo.create(tx, data)
  await outboxRepo.create(tx, {
    event_type: 'DOMAIN_CREATED',
    payload: { id: record.id },
    status: 'PENDING',
  })
})
```

Use Prisma's implicit per-query transaction (default) for single-model reads and writes.
Avoid long-held transactions — keep `$transaction` blocks as short as possible.
Timeout: set `timeout` option on interactive transactions if the operation may be slow.

## Query Optimization Rules

- Prevent N+1: always use `include` or `select` on relations needed in the response
- Large lists: use cursor-based pagination (`cursor: { id: lastId }, take: N, skip: 1`)
  — never `offset` + `limit` beyond [N] rows (causes full table scans at scale)
- Raw queries: use `db.$queryRaw<Type[]>` with Prisma.sql template literals; never
  string-concatenate SQL (SQL injection risk)
- Read-heavy endpoints: consider a Redis cache layer — see **backend-patterns skill**
- Explain plans: run `EXPLAIN ANALYZE` on any query touching > [N] rows before shipping

- Adding or updating OpenAPI/Swagger documentation
- Designing error response shapes

## Zod + Fastify Type Provider

Every route plugin must use `withTypeProvider<ZodTypeProvider>()` and declare a
`schema` object. This provides end-to-end TypeScript types at zero runtime cost.

```typescript
import { FastifyInstance } from 'fastify'
import { ZodTypeProvider } from 'fastify-type-provider-zod'
import { z } from 'zod'
import { CreateDomainSchema, DomainResponseSchema } from '../../schemas/domain.schema'
import * as domainService from '../../services/domain.service'

export async function domainRoutes(fastify: FastifyInstance) {
  const f = fastify.withTypeProvider<ZodTypeProvider>()

  f.post('/domains', {
    preHandler: [fastify.authenticate],
    schema: {
      body: CreateDomainSchema,
      response: { 201: DomainResponseSchema },
    },
  }, async (request, reply) => {
    const result = await domainService.create(fastify.db, request.body)
    return reply.status(201).send(result)
  })
}
```

## Zod Schema Rules

- Schema files live in `[backend_dir]/src/schemas/<domain>.schema.ts`
- Always export the Zod schema AND the inferred TypeScript type:
  ```typescript
  export const DomainSchema = z.object({ ... })
  export type Domain = z.infer<typeof DomainSchema>
  ```
- Request schemas (body, querystring, params): use `.strict()` to reject unknown fields
- Response schemas: use `.strip()` — never leak internal fields (password hash, internal flags)
- Every route declares error responses:
  ```typescript
  response: { 201: DomainResponseSchema, 400: ErrorSchema, 401: ErrorSchema, 404: ErrorSchema }
  ```
- Shared field patterns (UUID, ISO datetime, pagination cursor, sort order) go in
  `[backend_dir]/src/schemas/shared.schema.ts` — import from there; never duplicate

## OpenAPI Rules

[If `@fastify/swagger` is used:]
- Tag every route plugin with a domain tag: `schema: { tags: ['[domain]'], ... }`
- Server info, contact, and license defined once in `[backend_dir]/src/plugins/swagger.ts`
- Auth scheme (Bearer JWT) declared once in the swagger plugin — never per-route
- Generate and commit `openapi.json` as part of the build: `npm run build:openapi`

## API Versioning Rules

- URL prefix strategy: all routes under `/api/v[N]/...`
- Breaking changes (field removal, type change) require a new version prefix
- Non-breaking additions (new optional fields) may be shipped in the current version
- Deprecation notice: add `x-deprecated: true` to the route's OpenAPI `schema` before removal

## Error Response Shape

All error responses across the API conform to:
```typescript
{ error: string; code: string; details?: unknown }
```
The global Fastify error handler in `[backend_dir]/src/plugins/error-handler.ts` normalizes
ALL errors to this shape. Routes never construct error reply bodies directly.

Standard error codes:
- `VALIDATION_ERROR` — Zod parse failure (400)
- `UNAUTHORIZED` — missing or invalid JWT (401)
- `FORBIDDEN` — insufficient role (403)
- `NOT_FOUND` — resource does not exist (404)
- `CONFLICT` — duplicate or constraint violation (409)
- `INTERNAL_ERROR` — unhandled exception (500)

## File 9: `.claude/hooks/ts-diagnostics.py`

```python
#!/usr/bin/env python3
"""
PostToolUse hook: runs tsc --noEmit after any Edit/Write on a .ts or .tsx file.
Covers both frontend and backend since both are TypeScript.
Exits 2 on type errors so Claude is forced to address them immediately.
"""
import json
import os
import subprocess
import sys

data = json.load(sys.stdin)
file_path = (data.get("tool_input") or {}).get("file_path", "")

if not file_path.endswith((".ts", ".tsx")):
    sys.exit(0)

# Relative paths from the project root — filled in from project discovery
FRONTEND_DIR = "./[frontend_dir]"
BACKEND_DIR = "./[backend_dir]"


def is_under(child: str, parent: str) -> bool:
    """True only when child is strictly inside parent (avoids substring false-matches)."""
    c = os.path.normpath(os.path.abspath(child))
    p = os.path.normpath(os.path.abspath(parent))
    return c.startswith(p + os.sep) or c == p


if is_under(file_path, FRONTEND_DIR):
    check_dir = FRONTEND_DIR
elif is_under(file_path, BACKEND_DIR):
    check_dir = BACKEND_DIR
else:
    sys.exit(0)

if not os.path.isdir(check_dir):
    sys.exit(0)

result = subprocess.run(
    ["npx", "tsc", "--noEmit", "--pretty", "false"],
    cwd=check_dir,
    capture_output=True,
    text=True,
)

if result.returncode == 0:
    sys.exit(0)

print(f"TypeScript errors detected after editing: {file_path}")
print()
output = (result.stdout + result.stderr).splitlines()
print("\n".join(output[:50]))
sys.exit(2)
```
